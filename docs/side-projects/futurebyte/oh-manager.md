### 핵심 성과 및 의사결정 이력

#### 1. APIRoute 상속 Custom TransactionRoute 개발을 통한 트랜잭션 수명 주기 일괄 통제 (커넥션 누출 차단)
* **배경 & 제약사항**: 각 API 엔드포인트(라우터 함수)마다 DB 세션 관리(세션 시작, 커밋, 예외 시 롤백, 세션 종료) 코드가 반복적으로 들어가 코드의 가독성이 훼손되었습니다. 만약 특정 API에서 예외 처리 누수가 발생할 경우 DB 커넥션 풀이 고갈되거나 트랜잭션 오염이 생길 위험이 상존했습니다.
* **의사결정 (Why TransactionRoute 데코레이터?)**:
  - FastAPI의 APIRoute 클래스를 오버라이드하여, HTTP Request의 생성 및 Response 반환 라이프사이클을 가로채는 커스텀 **`TransactionRoute`**를 구현했습니다.
  - 이를 통해 API 호출 시작 시 세션을 바인딩하고, Response가 정상 반환되면 자동으로 커밋하며, 예외가 던져질 경우 자동으로 롤백을 수행하는 전역 미들웨어성 데코레이터 단에서 일괄 통제하도록 구조화했습니다.
* **성과**: 비즈니스 로직(Usecase/Service) 레이어에서 데이터베이스 영속성 트랜잭션에 관련된 코드를 완전히 제거하여 오직 순수 비즈니스 로직에만 집중하게 하였고, 커넥션 누수를 원천 차단했습니다.

#### 2. dependency-injector 기반 DI 패턴 도입으로 외부 SDK/인프라 변동 리스크 격리
* **배경 & 제약사항**: 오디션 매칭 도메인의 유스케이스 코드가 DB 라이브러리(SQLAlchemy)와 알림 발송용 외부 SDK(Slack SDK)에 밀결합되어 있어, 기술 스펙이 변경될 경우(예: Slack을 알림 전용 큐로 변경 등) 비즈니스 로직을 모두 뜯어고쳐야 하는 유지보수 리스크가 있었습니다.
* **의사결정 (Why Dependency Injector?)**:
  - `dataclasses`를 사용해 핵심 비즈니스 도메인 규칙을 보호하는 독립 엔티티를 구성했습니다.
  - 파이썬 환경의 고성능 DI 컴포넌트인 `dependency-injector` 컨테이너를 도입하여, 런타임에 인프라 어댑터(Slack 연동 모듈, DB 리포지토리)를 동적으로 주입하게 설정했습니다.
  - 테스트 환경에서는 `patch_ioc` 패턴을 적용하여 DB 연결 없이 Mock 리포지토리를 자동 바인딩하여 실행할 수 있도록 검증 환경을 구축했습니다.
* **성과**: 외부 프레임워크나 외부 API 규격이 변경되어도 오디션 코어 비즈니스 코드는 단 한 줄도 손대지 않는 구조적 유연성을 확보하고, 전체 테스트 커버리지를 비약적으로 높였습니다.
