### 핵심 성과 및 의사결정 이력

#### 1. Classical Mapping(map_imperatively)을 통한 DB와 도메인 의존성 단방향 격리 (DIP 실현)
* **배경 & 제약사항**: 일반적으로 테이블 데이터 모델 클래스를 도메인 모델로 혼용하지만, 이 경우 DB 스키마나 라이브러리 사양(SQLAlchemy 선언형 모델 등)이 변경되면 핵심 도메인 규칙이 직접 오염되는 취약점이 있었습니다. 이로 인해 DB 커넥션 없이는 핵심 로직 검증조차 어려워 테스트 신뢰성이 낮았습니다.
* **의사결정 (Why Classical Mapping?)**:
  - 도메인 중심 설계(DDD)를 실천하기 위해, 순수 비즈니스 엔티티 클래스와 SQLAlchemy의 테이블 매핑 정의를 물리적으로 분리하는 **Classical Mapping (`map_imperatively()`)** 방식을 도입했습니다.
  - 데이터 모델과 비즈니스 도메인 간의 의존성을 단방향(DIP)으로 끊음으로써 결합도를 차단했습니다.
* **성과**: 데이터베이스 의존성 없이 핵심 도메인의 단위 테스트 작성이 원활해져, 로컬 환경에서 빠른 피드백 루프를 수립하고 코드 안정성을 극대화했습니다.

#### 2. 동기식 Event Bus 구현을 통한 미디어 업로드와 리워드 트랜잭션 에러 전파 차단
* **배경 & 제약사항**: 미디어 업로드 로직과 리워드 지급 보상 로직이 단일 트랜잭션 내에 강하게 밀결합되어 있었습니다. 이 때문에 타사 리워드 지급 API 장애 시 전체 미디어 업로드 프로세스마저 함께 롤백되는 가용성 저하 문제가 발생했습니다.
* **의사결정 (Why Event Bus?)**:
  - 두 로직 간의 결합도를 제어하기 위해 자체 발행-구독(Pub-Sub) 패턴을 따르는 **동기식 Domain Event Bus**를 자체 구현했습니다.
  - 미디어 업로드 완료 시점에 이벤트를 발행하고, 리워드 지급기는 이벤트를 가로채 독립적인 트랜잭션으로 처리하도록 격리했습니다.
* **성과**: 미디어 업로드와 리워드 지급 로직 간의 물리적인 동기 결합을 제거하여 에러 전파를 차단했으며, 추후 알림 전송 등 새로운 리스너가 추가될 때 기존 핵심 코드를 고칠 필요 없이 이벤트 구독자만 추가하면 되는 확장성을 확보했습니다.
